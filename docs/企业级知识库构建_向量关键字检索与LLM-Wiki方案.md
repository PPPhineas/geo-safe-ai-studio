# 企业级知识库构建:向量检索 + 关键字检索 + LLM Wiki

## 1. 可行性结论

企业级知识库采用「向量检索 + 关键字检索 + LLM Wiki」是可行且推荐的路线。单纯 RAG 的主要问题是每次问答都从原始文档片段重新拼装知识,缺少长期沉淀;LLM Wiki 的价值在于把企业资料持续编译为结构化、互链、可审计的知识中间层。

本方案参考 Andrej Karpathy 的 llm-wiki 思路:

- 原始资料保持不可变,作为事实源。
- LLM 维护 Markdown/结构化 wiki,沉淀主题页、实体页、流程页、FAQ、冲突记录和决策记录。
- 检索时同时利用 wiki、原文 chunk、关键字索引和向量索引。
- 回答必须带来源、版本、引用位置和权限校验结果。

默认落地边界:

- 面向企业通用平台,而非单一业务模块。
- 私有化优先部署,模型、向量库、索引、文件存储与审计数据默认在内网或专有云。
- LLM Wiki 是受控派生层,不是唯一事实源;关键结论必须保留来源、置信度和审核状态。

## 2. 总体架构

### 2.1 原始资料层

原始资料层保存企业知识的事实源,包括制度文档、项目资料、会议纪要、FAQ、工单、代码文档、网页归档、表格、图片、扫描件等。

关键要求:

- 原文只读保存,不得被 LLM 改写。
- 保存文档 ID、来源系统、作者、部门、版本、时间、密级、权限范围、hash。
- 支持重建索引和重跑 wiki ingest。
- 同一文档多版本并存,检索时默认使用最新有效版本。

### 2.2 混合检索层

混合检索层包含向量索引、BM25/关键字索引、元数据过滤和 rerank。

职责:

- 向量检索负责语义召回。
- BM25/关键字检索负责精确术语、编号、制度条款、项目名、人名、设备名和专业缩写。
- 元数据过滤负责权限、部门、密级、时间、文档类型、版本状态。
- reranker 对候选结果统一排序,提高最终上下文质量。

推荐技术选型:

- 向量库:Milvus、Qdrant、pgvector、Elasticsearch/OpenSearch kNN 均可。
- 关键字检索:Elasticsearch/OpenSearch、PostgreSQL full text、Lucene 系引擎。
- 文档与元数据:PostgreSQL 或企业已有内容管理系统。
- 文件存储:对象存储、NAS 或企业文档平台。

### 2.3 LLM Wiki 层

LLM Wiki 是原文资料之上的可审计派生知识层。它不是聊天记录,也不是一次性摘要,而是一组长期维护的结构化页面。

典型页面类型:

- `overview`:领域或业务概览。
- `entity`:客户、项目、系统、设备、团队、角色等实体页。
- `topic`:主题页或专题综述。
- `process`:流程、制度执行路径、操作规程。
- `faq`:高频问题与标准答复。
- `decision`:决策记录和原因。
- `conflict`:冲突、过期、待复核信息。
- `source_summary`:单篇来源文档摘要。

推荐 frontmatter:

```yaml
---
title: "页面标题"
type: "topic"
owners: ["部门或角色"]
source_ids: ["doc-001", "doc-002"]
confidence: "medium"
review_status: "pending"
last_verified_at: "2026-06-18"
sensitivity: "internal"
---
```

### 2.4 治理与审计层

企业级知识库必须把治理作为一等能力,否则 LLM 产出会逐步积累幻觉、过期结论和权限风险。

治理能力:

- SSO/RBAC/ABAC 权限控制。
- 文档级、段落级、wiki 派生结论级权限继承。
- LLM 生成内容审核流。
- 敏感信息识别、脱敏和外发阻断。
- 全链路审计:导入、解析、检索、问答、wiki 更新、人工审核。
- 知识健康检查:无来源断言、孤立页面、过期制度、冲突结论、缺失反链。

## 3. 核心流程

### 3.1 文档入库

1. 从上传、同步连接器或 API 接收文档。
2. 抽取元数据并计算 hash。
3. 解析正文、表格、图片 OCR、附件和标题层级。
4. 按语义结构切分 chunk,保留章节路径和页码位置。
5. 写入文档库、BM25 索引和向量索引。
6. 记录解析与索引状态。

### 3.2 Wiki Ingest

1. 对新文档生成 `source_summary`。
2. 根据实体、主题、流程和术语识别候选 wiki 页面。
3. 更新相关页面,输出结构化 diff。
4. 标记新增事实、被修订事实、冲突事实和待复核项。
5. 写入操作日志,记录来源文档、影响页面、模型版本和审核状态。
6. 进入自动发布、异常审核或人工审核流程。

### 3.3 查询问答

1. 识别用户身份、权限、问题意图和约束条件。
2. 并行检索 wiki 页面、原文 chunk、关键字索引和结构化元数据。
3. 对候选结果做权限裁剪和 rerank。
4. 优先使用已审核 wiki;wiki 不足时回查原文 chunk。
5. 生成答案,引用来源文档、版本、章节或页码。
6. 对无来源、权限不足、信息冲突和低置信度场景显式说明。

### 3.4 Wiki Lint

定期运行知识库健康检查:

- 无来源断言。
- 过期制度或被新版本替代的内容。
- 互相冲突但未标记的结论。
- 孤立页面和缺失反链。
- 重要概念被频繁提及但没有独立页面。
- 低置信度页面长期未审核。
- 用户高频问题没有沉淀为 FAQ 或专题页。

## 4. 核心接口建议

```http
POST /documents/import
```

导入文档,返回文档 ID、解析状态和索引状态。

```http
POST /wiki/ingest
```

对指定文档或文档集合生成/更新 wiki,返回变更集和审核状态。

```http
POST /search
```

执行混合检索,返回 wiki 命中、原文 chunk、关键词命中、rerank 分数和权限裁剪信息。

```http
POST /answer
```

基于检索结果生成带引用回答。

```http
GET /wiki/pages/{id}
```

查看 wiki 页面、来源、状态、关系和历史版本。

```http
POST /wiki/review/{change_id}
```

审核 wiki 变更,支持通过、驳回、要求重写。

## 5. 分阶段实施计划

### Phase 1:PoC 闭环

目标是验证混合检索和引用问答质量。

- 接入 2 到 3 类企业资料源,例如制度文档、项目文档、FAQ。
- 完成文档解析、chunk、embedding、BM25、混合检索。
- 实现带引用回答,禁止无来源编造。
- 评估向量检索、关键字检索和混合检索的命中质量。

验收标准:

- 精确编号、制度条款、项目名可由关键字检索稳定命中。
- 语义改写问题可由向量检索命中。
- 混合检索整体优于单独 BM25 或单独向量检索。
- 回答能稳定返回来源文档、版本和引用位置。

### Phase 2:LLM Wiki

目标是让知识库从一次性 RAG 升级为可持续沉淀的知识系统。

- 定义 wiki 页面模板和 frontmatter。
- 实现 ingest agent,根据新文档更新相关页面。
- 对 wiki 更新输出可审核 diff。
- 实现冲突标记、过期标记和待复核项。
- 实现 wiki lint agent,定期生成健康检查报告。

验收标准:

- 新文档 ingest 后生成 `source_summary`,并更新相关主题页或实体页。
- 冲突信息不会被静默覆盖,必须形成冲突说明或待审核项。
- 人工驳回的 wiki 变更不会进入默认检索结果。
- wiki 页面中的关键断言能追溯到原始文档。

### Phase 3:企业治理

目标是满足企业内控、权限、审计和长期运营要求。

- 接入 SSO/RBAC,检索和生成阶段都做权限裁剪。
- 加入文档密级、部门、版本和有效期规则。
- 加入敏感信息识别、脱敏和外发阻断。
- 建立审计日志和运营指标。
- 建立质量看板:引用覆盖率、无来源回答率、人工驳回率、wiki 过期率、检索命中率。

验收标准:

- 不同权限用户无法看到无权原文或由其派生的 wiki 结论。
- 查询、检索、回答和 wiki 更新均可审计。
- 过期内容、新旧版本冲突和低置信度内容可被监控。

## 6. 测试场景

### 6.1 检索测试

- 输入制度编号,必须命中对应制度原文。
- 输入项目简称,必须召回项目全称、相关实体页和原始项目资料。
- 输入语义改写问题,必须召回同义主题资料。
- 输入跨文档综合问题,必须同时召回 wiki 综述和多个原文 chunk。

### 6.2 回答测试

- 知识库无覆盖时,必须提示未找到可靠来源。
- 权限不足时,不得泄露原文内容、标题中的敏感信息或派生结论。
- 文档存在新旧版本时,默认引用最新有效版本。
- 多来源冲突时,必须展示冲突,不得强行合并为单一结论。

### 6.3 Wiki 测试

- 新文档进入后,必须生成摘要页并记录日志。
- 更新已有主题页时,必须保留来源 ID。
- 删除或驳回 wiki 变更后,检索结果不应继续使用该变更。
- lint 能发现无来源断言、孤立页面和过期页面。

## 7. 风险与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| LLM 幻觉 | wiki 页面出现无来源断言 | 强制来源字段、审核状态、lint 检查 |
| 权限泄露 | 派生 wiki 泄露无权原文内容 | 派生内容继承来源权限,检索和生成双重裁剪 |
| 版本混乱 | 新旧制度同时被引用 | 文档版本状态和有效期进入元数据过滤 |
| 检索噪声 | 向量召回语义相近但不相关内容 | BM25 + 元数据过滤 + reranker |
| 维护漂移 | wiki 分类、关系、术语逐渐不一致 | schema 固化、受控词表、定期 lint |
| 成本失控 | 大量文档反复 ingest 和 rerank | 增量索引、hash 去重、批处理和缓存 |

## 8. 默认假设

- 企业知识库默认部署在内网或专有云。
- 原始文档是事实源,wiki 是可审计、可重建、可审核的派生知识层。
- 所有回答默认需要引用来源;无来源时不得编造。
- 权限控制必须作用于原文、检索结果、wiki 页面和最终回答。
- 先做平台通用能力,再通过业务域 schema、模板、术语表和权限规则扩展到具体部门。
